Payment Abstraction Architecture
This is the most architecturally significant decision in this sprint. The current BillingClient is Stripe-only with mock fallback. It needs to become a provider-agnostic interface.
Design principle: The provisioning pipeline only cares about two outcomes — payment_confirmed and payment_failed. It doesn't care which provider produced that outcome. The webhook handler is the convergence point; everything above it is provider-specific.
Gateway Interface
Define an abstract base in a new file services/payment/base.py:
pythonfrom __future__ import annotations
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Any

@dataclass
class CheckoutResult:
    session_id: str
    checkout_url: str
    customer_ref: str       # provider's customer ID or reference
    provider: str           # "stripe" | "dpo"
    mock_mode: bool = False

@dataclass
class WebhookEvent:
    event_type: str         # "payment.confirmed" | "payment.failed" | "subscription.cancelled"
    tenant_id: str | None
    subscription_id: str | None
    customer_ref: str | None
    raw: dict[str, Any]

class PaymentGateway(ABC):
    @abstractmethod
    def create_checkout(self, tenant, owner) -> CheckoutResult: ...

    @abstractmethod
    def parse_webhook(self, payload: bytes, headers: dict[str, str]) -> WebhookEvent: ...

    @property
    @abstractmethod
    def provider_name(self) -> str: ...

Gateway Middleware / Factory
services/payment/factory.py:
pythonfrom app.config import get_settings
from app.services.payment.base import PaymentGateway
from app.services.payment.stripe_gateway import StripeGateway
from app.services.payment.dpo_gateway import DPOGateway

_REGISTRY: dict[str, type[PaymentGateway]] = {
    "stripe": StripeGateway,
    "dpo": DPOGateway,
}

def get_payment_gateway() -> PaymentGateway:
    provider = get_settings().active_payment_provider.lower()
    cls = _REGISTRY.get(provider)
    if cls is None:
        raise ValueError(f"Unknown payment provider: '{provider}'. Valid: {list(_REGISTRY)}")
    return cls()
Adding a third provider in the future (Flutterwave, Pesapal, PayStack) is a single file + one line in _REGISTRY. Nothing else changes.
Webhook Middleware
The existing /billing/webhook endpoint is Stripe-specific — it checks for Stripe-Signature. This needs to become provider-aware. The cleanest approach is a single webhook endpoint that delegates to the active gateway:
python# routers/billing.py
gateway = get_payment_gateway()

@router.post("/webhook/{provider}", response_model=MessageResponse)
async def billing_webhook(provider: str, request: Request, ...):
    if provider != gateway.provider_name:
        raise HTTPException(400, f"This instance uses {gateway.provider_name}")
    
    payload = await request.body()
    try:
        event = gateway.parse_webhook(payload, dict(request.headers))
    except ValueError as exc:
        raise HTTPException(400, str(exc))
    
    # From here: pure business logic, provider-agnostic
    if event.event_type == "payment.confirmed":
        ...
    elif event.event_type == "payment.failed":
        ...
    elif event.event_type == "subscription.cancelled":
        ...
```

The Stripe webhook URL becomes `/billing/webhook/stripe`. The DPO notify URL becomes `/billing/webhook/dpo`. Traefik routes both. Stripe dashboard and DPO merchant portal each get their respective endpoint.
```
