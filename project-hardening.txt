1. Executive Summary
The ERP SaaS platform has achieved its MVP milestone: sites can be provisioned inside the 
existing ERPNext Docker bench, admin passwords reset, and the control plane is accessible 
through Traefik. The core machinery works.
This document is a complete engineering roadmap to harden that foundation to enterprise 
grade. It covers nine critical domains — security, observability, resilience, UI/UX, billing, multi-
tenancy enforcement, data integrity, DevOps, and documentation — with precise, actionable 
specifications for each.

Current Baseline
Site creation functional. Admin password reset confirmed. JWT auth in place. PostgreSQL + Redis + 
RQ worker wired. Next.js dashboard scaffold live.

Hardening Goal
Rate-limited, audited, idempotent API. Stripe billing with plan enforcement. Real-time job tracking. 
Polished UI with landing page. Alembic migrations. Structured logging. CI/CD with health gates.

3. Security Hardening
3.1 Database Migrations — P0
The current pattern of Base.metadata.create_all(bind=engine) on startup is unsafe for 
production. Any model change can silently corrupt data or be ignored. Alembic must be 
introduced immediately.
Required changes
•	Add alembic==1.16.5 to requirements.txt (already present — wire it)
•	Create alembic/ directory with env.py pointing to app.db.Base.metadata
•	Generate initial migration from current models
•	Replace create_all call in main.py with alembic upgrade head subprocess at startup
•	Add CI step: alembic check — fails if unapplied migrations exist

# alembic/env.py (key excerpt)
from app.models import Base   # import all models so they're visible
target_metadata = Base.metadata

# provisioning-api/app/main.py startup
import subprocess
@app.on_event('startup')
def startup():
    subprocess.run(['alembic', 'upgrade', 'head'], check=True)

3.2 Rate Limiting — P0
Without rate limiting, the /auth/login and /auth/signup endpoints are trivially brute-forced. The 
/tenants POST endpoint can be abused to exhaust bench capacity. Use slowapi (the FastAPI-
native Starlette limiter).
Limits to enforce
•	POST /auth/login — 10 requests / minute / IP
•	POST /auth/signup — 5 requests / minute / IP
•	POST /tenants — 3 requests / minute / user
•	POST /tenants/{id}/backup — 1 request / 5 minutes / tenant
•	All other authenticated endpoints — 60 requests / minute / user

# provisioning-api/requirements.txt — add:
slowapi==0.1.9

# app/main.py
from slowapi import Limiter, _rate_limit_exceeded_handler
from slowapi.util import get_remote_address
from slowapi.errors import RateLimitExceeded

limiter = Limiter(key_func=get_remote_address, storage_uri=settings.redis_url)
app.state.limiter = limiter
app.add_exception_handler(RateLimitExceeded, _rate_limit_exceeded_handler)

# app/routers/auth.py
@router.post('/login')

3.3 JWT Refresh Tokens & Revocation — P0
The current implementation issues 24-hour access tokens with no revocation. A stolen token is 
valid for a full day. The fix is to shorten access token lifetime and add a Redis-backed 
revocation list.
Token lifecycle
•	Access token: 15-minute expiry — short enough to limit stolen-token damage
•	Refresh token: 7-day expiry — stored in HttpOnly cookie, not localStorage
•	Revocation: on logout, store jti in Redis with TTL matching remaining token lifetime
•	Middleware: on every request, check jti against Redis revocation set

# app/security.py — new constants
ACCESS_TOKEN_EXPIRE_MINUTES  = 15
REFRESH_TOKEN_EXPIRE_DAYS    = 7

# app/routers/auth.py — logout endpoint
@router.post('/logout')
def logout(credentials: ..., redis: Redis = Depends(get_redis)):
    payload = decode_access_token(credentials.credentials)
    jti = payload['jti']
    ttl = int(payload['exp'] - time.time()) + 60
    redis.setex(f'revoked:{jti}', ttl, '1')
    return {'message': 'Logged out'}

# app/deps.py — revocation check
def get_current_user(credentials, db, redis):
    payload = decode_access_token(credentials.credentials)
    if redis.exists(f"revoked:{payload['jti']}"):
        raise HTTPException(401, 'Token revoked')

3.4 Security Headers & CORS — P0
The API and UI currently ship no security headers. This leaves them exposed to XSS, 
clickjacking, and cross-origin data exfiltration. Add a hardened middleware layer.

# app/middleware/security.py
from starlette.middleware.base import BaseHTTPMiddleware

class SecurityHeadersMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        response = await call_next(request)
        response.headers['Strict-Transport-Security'] = 'max-age=31536000; 
includeSubDomains'
        response.headers['X-Content-Type-Options']    = 'nosniff'
        response.headers['X-Frame-Options']           = 'DENY'
        response.headers['Referrer-Policy']           = 'strict-origin-when-cross-
origin'
        response.headers['Content-Security-Policy']   = "default-src 'self'"
        return response

# app/main.py — CORS (whitelist only)
from fastapi.middleware.cors import CORSMiddleware
app.add_middleware(CORSMiddleware,
    allow_origins=[settings.saas_ui_origin],
    allow_credentials=True,
    allow_methods=['GET','POST','DELETE'],
    allow_headers=['Authorization','Content-Type'],
)

4. Billing & Plan Enforcement
4.1 Stripe Integration — P1
The current billing_client.py registers a Customer in the platform ERP site but has no payment 
capture, subscription lifecycle, or plan enforcement. Until Stripe is wired, the platform cannot 
collect revenue and plans are meaningless.
Stripe objects required
•	Product: 'ERPNext SaaS' — one product, three Price objects 
(starter/business/enterprise)
•	Customer: created on signup, stored as stripe_customer_id on the User model
•	Subscription: created when tenant provisioning is confirmed — status determines 
tenant.status
•	Checkout Session: hosted by Stripe, redirect back to /dashboard after payment
•	Webhook: stripe.Webhook.construct_event verifies signature on /billing/webhook

# Stripe price IDs in .env.example
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...
STRIPE_PRICE_STARTER=price_...
STRIPE_PRICE_BUSINESS=price_...
STRIPE_PRICE_ENTERPRISE=price_...

Provisioning gate
Tenant provisioning must be gated on payment. The flow must be:
1.	User submits create-tenant form ? API returns a Stripe Checkout Session URL
2.	User completes payment on Stripe-hosted page
3.	Stripe fires checkout.session.completed webhook to /billing/webhook
4.	Webhook handler enqueues the RQ provisioning job
5.	Tenant status transitions: pending ? provisioning ? active

Critical
Never start site provisioning before webhook payment confirmation. Enqueue only from the webhook 
handler, never from the tenant create endpoint directly.

5. Worker Resilience & Job Tracking
5.1 Job Retry Logic — P1
The current RQ worker has no retry policy. A bench timeout (e.g., MariaDB is briefly 
unavailable) causes a permanent job failure with no recovery. Add structured retries with 
exponential backoff.

# app/services/tenant_service.py — add retry config
from rq.job import Retry

rq_job = queue.enqueue(
    'app.workers.tasks.provision_tenant',
    job.id, tenant.id, owner.email, admin_password,
    job_timeout='10m',
    retry=Retry(max=3, interval=[30, 120, 300]),  # 30s, 2m, 5m
)

5.2 Dead Letter Queue — P1
After all retries are exhausted, failed jobs must land in a dead-letter queue for manual inspection 
rather than vanishing. An admin API endpoint must expose these jobs.

# app/queue/enqueue.py — DLQ queue
def get_dlq() -> Queue:
    return Queue('dead-letter', connection=get_redis_connection())

# RQ worker on_failure callback
def on_job_failure(job, connection, type, value, traceback):
    dlq = get_dlq()
    dlq.enqueue('app.workers.dlq.handle_failed', job.id, job.args)

5.3 Real-Time Job Progress via WebSocket — P1
Users currently have no way to see provisioning progress without polling the job endpoint. A 
WebSocket endpoint allows the dashboard to display live log lines as they stream from the 
worker.

# app/routers/ws.py
from fastapi import WebSocket

@router.websocket('/ws/jobs/{job_id}')
async def job_stream(websocket: WebSocket, job_id: str, token: str):
    await websocket.accept()
    # Subscribe to Redis pub/sub channel: job:{job_id}:logs
    # Worker publishes each log line to this channel
    async for message in redis_pubsub(f'job:{job_id}:logs'):
        await websocket.send_text(message)
        if message == '__DONE__': break
    await websocket.close()

8.3 Backup System Hardening — P2
Backups are currently fire-and-forget with no manifest, no retention policy, and no off-server 
copy. The hardening plan:
•	Create a backups table: id, tenant_id, job_id, file_path, file_size_bytes, created_at, 
expires_at, s3_key
•	After successful bench backup, parse output to capture backup file path
•	Write record to backups table with expiry based on plan (7/30/90 days)
•	Schedule nightly cron job (RQ scheduled job) to delete expired backup files and DB 
records
•	Add S3 upload step (boto3) — optional, enabled by BACKUP_S3_BUCKET env var
•	Expose GET /tenants/{id}/backups endpoint returning backup manifest

9.3 Health Check Improvements — P2
The current GET /health returns a static response. A real health check must verify all 
dependencies:

@app.get('/health')
async def health(db: Session = Depends(get_db), redis = Depends(get_redis)):
    checks = {}
    try:
        db.execute(text('SELECT 1'))
        checks['postgres'] = 'ok'
    except Exception as e:
        checks['postgres'] = f'error: {e}'
    try:
        redis.ping()
        checks['redis'] = 'ok'
    except Exception as e:
        checks['redis'] = f'error: {e}'
    status = 'ok' if all(v == 'ok' for v in checks.values()) else 'degraded'
    code = 200 if status == 'ok' else 503
    return JSONResponse({'status': status, 'checks': checks}, status_code=code)

13. Definition of Done — Enterprise Readiness
The platform is considered enterprise-ready when ALL of the following are true:

•	All P0 security items shipped and verified in production
•	Stripe payments operational end-to-end in live mode
•	Zero sites provisioned without successful payment confirmation
•	Alembic migrations in place — zero create_all() calls remaining
•	Rate limiting active on all public endpoints — verified via load test
•	JWT tokens expire in ?15 minutes — revocation tested
•	Audit log captures every state-changing action
•	Sentry capturing errors from both API and worker processes
•	Provisioning failure sends email to tenant owner within 60 seconds
