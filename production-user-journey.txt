1. Executive Summary
Biashara Cloud is a multi-tenant ERP SaaS platform targeting SME and mid-market businesses operating in Tanzania. The current codebase implements the core happy-path: signup, onboarding, tenant creation, and basic admin oversight. This document maps the complete production user journey, captures all missing functional requirements, edge cases, failure and recovery flows, and operational features required before a general-availability launch.
The analysis is grounded in the existing Next.js 15 frontend (saas-ui), the API contract inferred from lib/api.ts, and the product positioning as a Tanzania-first, mobile-friendly, TZS-aware business operations platform.
2. Complete User Journey Map
The platform serves three distinct actor types, each with a distinct journey.

4. Edge Cases
4.1 Subdomain & Identity Conflicts
•	User types subdomain, backend returns 409 (conflict): show inline error 'That subdomain is already taken' and clear the field.
•	Subdomain passes client validation but fails on backend due to DNS reservation (e.g., 'www', 'api', 'mail'): backend must maintain a blocklist; frontend must surface the message verbatim.
•	Two concurrent onboarding tabs for the same user submitting identical subdomains: idempotency key (X-Idempotency-Key header already implemented) must deduplicate; second request should return the same tenant.
•	Company name contains special characters (Arabic, Swahili diacritics): subdomain sanitizer already strips non-[a-z0-9-] chars, but company_name must accept Unicode up to 255 chars.
4.2 Payment & Checkout Edge Cases
•	User opens checkout in new tab, completes payment, then closes tab: on returning to the original tab (still on 'waiting' step), polling should pick up the status change within 5 s and advance.
•	Checkout session expires (Stripe: 24h, DPO: varies): display 'Your payment link has expired' with a 'Generate new link' button that calls POST /tenants/:id/checkout/renew.
•	Payment succeeds but webhook is delayed >60 s: user presses 'I already completed payment'; frontend moves to polling. Backend must reconcile payment state asynchronously.
•	Duplicate payment: user pays twice due to back-navigation. Checkout provider must enforce idempotency; backend must reject second charge and return the same checkout_session_id.
•	Currency conversion: DPO charges in TZS, Stripe in USD. Displayed TZS estimate on plan cards must note it is approximate and subject to exchange rate at time of charge.
4.3 Provisioning Edge Cases
•	Provisioning succeeds partially (app installed but SSL cert pending): tenant status may be 'active' but domain not yet reachable. Need sub-status or readiness probe endpoint GET /tenants/:id/readiness.
•	Provisioning worker crashes mid-run, job stuck in 'running' state with no progress for >10 min: backend dead-letter timeout should move job; frontend should surface 'Provisioning is taking longer than expected – contact support' after 15 min of polling with no state change.
•	chosen_app value not supported by current backend version: existing retry-without-chosen_app logic handles 422, but user sees no explanation. Add a soft warning: 'The selected business focus was not applied; your workspace defaults to Core Operations.'
•	Subdomain normalisation divergence: frontend normalises subdomain to [a-z0-9-] but backend applies different rules. The preview domain shown at step 1 may differ from the actual provisioned domain. Backend should return the canonical domain in the tenant object (already done) and the success screen must show tenant.domain, not the locally-computed value.
•	ERP app first-boot latency: newly provisioned ERPNext can take 3-5 min after status='active' before the HTTP endpoint responds. Workspace URL should open in a new tab with a friendly loading page, not immediately 404.
4.4 Session & Authentication Edge Cases
•	JWT expires while user is filling out the onboarding form: on submit, API returns 401. Currently the error is displayed as a generic failure. Must detect sessionExpired, save form state to sessionStorage, redirect to /login?next=/onboarding, and restore form state on return.
•	User has multiple browser tabs open; token is cleared in one tab (logout): SESSION_EXPIRED_EVENT broadcast via custom window event already implemented but only listened to in DashboardPage. Must be applied globally (e.g., in a root layout effect or in middleware).
•	Token with role='admin' is used but the ROLE_COOKIE has expired or been cleared: middleware falls back to decoding the token payload directly (already implemented), but ROLE_COOKIE and TOKEN_COOKIE TTLs must stay in sync. If they diverge, the nav link may not appear while the page still loads.
•	Admin route accessed with a valid but non-admin token: middleware correctly returns 403 HTML page. But direct navigation via the browser address bar shows a raw HTML error page, not the branded 403. The forbiddenHtml() helper should reference the design system styling.
4.5 Network & Connectivity Edge Cases
•	Low-bandwidth connection (common on 3G/4G in Tanzania): large bundle sizes cause slow TTFB. Consider code splitting at route level, lazy-loading admin page, and adding a loading skeleton for the tenant table.
•	WebSocket (job log streaming) not available (corporate proxy, firewall): ws.onerror handler already falls back to polling – this path must be tested. Polling interval should back off exponentially (5s → 10s → 30s) for terminal states.
•	User offline mid-operation: fetch throws NetworkError. The request helper wraps fetch but does not distinguish network failures from API errors. Add a navigator.onLine check and an 'You appear to be offline – changes will not be saved' banner.
•	API base URL resolveApiBase() guard: prevents localhost API calls from production browser context. Edge case: internal VPN user accessing production UI from a machine where localhost:PORT is running a dev API. The guard correctly falls back to /api in this case – verify reverse proxy is configured to handle /api prefix.
5. Failure & Recovery Flows
5.1 Failure Classification
Category	Example	Detection	Owner
Transient API failure	500 on POST /tenants (DB timeout)	HTTP 5xx response	Frontend retry + user retry CTA
Permanent API failure	422 invalid payload	HTTP 4xx response	Frontend form validation improvement
Payment failure	Card declined / checkout timeout	webhook / polling	User self-service via billing portal
Provisioning failure	Docker container OOM / DNS failure	Job status = 'failed'	Ops alert + user retry + admin intervention
Partial provisioning	App installed, SSL pending	Readiness probe	Auto-retry SSL cert; show status to user
Job stream failure	WebSocket disconnect mid-stream	ws.onerror	Silent fallback to polling (already coded)
Session expiry	JWT exp reached	401 response / JWT decode	Re-auth redirect preserving intent
Network offline	No connectivity	NetworkError / navigator.onLine	Offline banner, queue retry on reconnect
Data inconsistency	Tenant 'active' but ERP not responding	Readiness probe / 404 on workspace URL	Automated health check + ops alert

5.2 Detailed Recovery Flows
RF-01: Provisioning Failure (User Flow)
23.	Polling detects status = 'failed'.
24.	Progress bar turns red; error message displays backend error string (or generic fallback).
25.	'Retry provisioning' button calls POST /tenants/:id/retry → creates new job.
26.	If retry also fails after 2 attempts, button changes to 'Contact support' linking to support channel.
27.	Admin is notified via alert (RF-05) regardless of user action.
RF-02: Interrupted Onboarding (User Returns)
28.	User returns to /onboarding after closing the tab.
29.	Frontend checks for persisted tenantId in localStorage (set at submitTenant).
30.	If found, calls GET /tenants/:id to restore current status.
31.	Step is restored: 'payment' if checkout_url exists and status=pending_payment; 'waiting' if provisioning; 'success' if active.
32.	If tenantId is stale or deleted, localStorage entry is cleared and wizard resets to step 1.
RF-03: Payment Webhook Delay
33.	User completes payment on DPO/Stripe; webhook has not arrived.
34.	User clicks 'I already completed payment'; step advances to 'waiting'.
35.	Polling begins. Backend reconciles payment within 60 s (async job).
36.	If not reconciled after 5 min, polling shows 'Payment verification taking longer than expected – please contact support if you were charged.'
37.	Once webhook arrives, backend updates status → polling picks up within 5 s.
RF-04: Admin Password Reset – Credential Visibility
38.	Admin clicks 'Reset admin login'; confirmation modal requires typing SUBDOMAIN in uppercase.
39.	On confirm, result.admin_password is displayed for 30 s (countdown timer already implemented).
40.	Copy-to-clipboard button with visual feedback.
41.	After 30 s, credential is zeroed from DOM; user must reset again if needed.
42.	All resets are written to audit log (ADM-03).
RF-05: Ops Alert Routing (New)
43.	Backend emits structured events: provisioning_failed, job_stuck (>15 min), payment_webhook_missed (>10 min).
44.	Alert service sends email/Slack notification to ops team with tenant_id, company_name, and error snippet.
45.	Admin Control Center 'Attention lane' badge turns amber/red and shows count of unresolved alerts.
46.	Admin can mark alert as 'acknowledged' or 'resolved' from the panel.
5.3 Error Message Standards
All user-facing errors must follow this hierarchy:
47.	Specific actionable message: 'Subdomain 'acme' is already taken. Please choose a different one.'
48.	Categorised fallback: 'Provisioning failed. Please try again or contact support.'
49.	Generic last resort: 'Something went wrong. Please refresh and try again.' (never expose raw stack traces)
Backend error strings (detail / message / error fields) may be surfaced directly only if they are user-safe. Engineering must maintain a list of safe-to-surface error codes.
