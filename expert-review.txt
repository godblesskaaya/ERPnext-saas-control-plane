Your SaaS is not “ERPNext with a UI on top.”
It is a platform with several bounded domains:

identity and access

tenant lifecycle / provisioning

tenant operations / management

billing and subscriptions

support and internal admin

observability / audit / compliance

platform configuration / policy

If these concerns get mixed, the system becomes messy fast.
So let’s separate them properly.

1. What the control plane actually is

The control plane is the system that manages customer ERP instances.

It should not contain tenant business data like invoices, stock, HR, payroll from customer tenants. That belongs inside each tenant ERPNext site.

The control plane owns:

who the customer is

what plan they are on

what site belongs to them

whether the site exists, is suspended, active, or deleted

when billing renews

what operations are allowed

what backups exist

what custom domains are mapped

what audit trail exists

what support actions were taken

So the control plane is the platform brain.

2. Top-level domain decomposition

A good separation is this:

A. Identity & Access

Who is using the SaaS control plane, and what can they do?

B. Tenant Registry

The source of truth for tenants and instances.

C. Provisioning

Creates, updates, upgrades, suspends, restores, and deletes ERPNext sites.

D. Tenant Management

Allows tenant admins to operate their instance after provisioning.

E. Billing & Subscription

Owns plans, pricing, invoices, payments, renewals, dunning, suspension triggers.

F. Platform ERP Integration

Uses your internal ERPNext site for finance, invoicing, CRM, and internal ops.

G. Support & Admin Operations

Tools for your team to inspect, assist, or intervene.

H. Audit, Monitoring & Policy

Tracks actions, platform health, quotas, and policy enforcement.

That’s the clean structure.

Important design note

This identity layer is for the SaaS portal, not for the tenant ERP itself.

A tenant admin may have:

one account in the SaaS control plane

another admin user inside their ERPNext tenant

Those are separate systems, even if linked.

5. Provisioning domain

This is the automation engine.

In your setup, provisioning is centered around bench new-site, but the domain is broader than just that command.

Responsibilities

create site

install apps

apply default settings

create default users

seed initial config

bind the site to the customer account

run health checks

mark site as ready

retry or roll back on failure

Provisioning workflow

A proper workflow looks like this:

Step 1: Request validation

Check:

subdomain availability

plan eligibility

billing state

customer email verified

domain policy rules

Step 2: Job creation

Create a provisioning job record.

Step 3: Site creation

Run:

bench new-site

install-app erpnext

any required custom apps

Step 4: Bootstrap

Optional setup:

company creation wizard

locale/timezone defaults

starter data

API user or integration keys

feature flags based on plan

Step 5: Health verification

Check:

site resolves by host

login page loads

database connection works

workers are healthy

Step 6: Finalization

Mark tenant active and notify customer.

Provisioning sub-features

This domain should also own:

Re-provision / retry failed setup

For failed jobs.

Clone environment

Later useful for staging/demo.

Upgrade app set

Install or remove apps based on plan.

Change runtime placement

Move tenant to another bench/node later.

Domain attach

Add verified custom domains.

Deprovision

Gracefully suspend, archive, or delete.

Important architectural rule

Provisioning must be job-based, not request-thread based.

The API should create jobs, and workers should perform the actual operations.

That gives you:

retries

logs

progress tracking

failure handling

non-blocking UI

Important relationship with provisioning

Billing should not directly run shell actions.

Instead, billing emits entitlement decisions like:

allow provisioning

suspend tenant

unsuspend tenant

restrict new backups

deny upgrades

Provisioning or policy modules then execute operational effects.

That separation avoids chaos.

8. Platform ERP integration domain

This is where your dedicated internal ERPNext site comes in.

This site should be treated as the system of record for commercial and internal business operations, not as the technical control plane database.

That distinction is important.

Use the Platform ERP for:

CRM leads/customers

quotations if needed

sales invoices

payment entries

subscriptions

finance reporting

receivables

taxes

support or internal ops workflows if useful

Do not use Platform ERP as:

the only technical tenant registry

the job queue

the source of runtime state

the operational event store

the only place that knows site status

Because operational state changes fast and needs strict technical modeling.

So the right pattern is:

Control plane database

System of record for technical platform state.

Platform ERP site

System of record for financial/commercial business records.

Integration examples

When a user signs up:

create customer/account in control plane

optionally create Customer record in Platform ERP

When subscription is activated:

create/update Subscription in Platform ERP

generate invoice there

When payment is confirmed:

billing module updates control plane entitlement

Platform ERP records invoice/payment

When tenant is suspended for non-payment:

control plane marks tenant suspended_billing

Platform ERP stores the financial trail

So this is an integration boundary, not a collapse into one system.

Very useful:

This can be inside your control plane, or partly inside Platform ERP if you prefer.

Separation rule

Support tools must be scoped and auditable.

Every support action should produce an audit log:

new state

Policy engine should cover:

Why explicit policy matters

Without explicit policy, business rules end up scattered:

some in UI

some in billing

some in API

some in scripts

That gets dangerous fast.

A policy layer centralizes platform rules.

16. What belongs where

This is where systems often go wrong, so let’s be explicit.

Belongs in the control plane

tenant metadata

subscription status

operational site status

provisioning jobs

lifecycle state

backup catalog

domain mappings

memberships

audit trails

Belongs in Platform ERP

customer financial documents

sales invoices

subscription accounting

payment entries

receivables

revenue reporting

Belongs in customer tenant ERP

their business data

their users

their invoices, items, HR, stock, etc.

This three-way separation is very healthy.

Design principles to keep the system clean

A few rules will save you a lot of pain.

Principle 1

Technical truth and financial truth are not the same thing.

Control plane DB = technical truth
Platform ERP = financial/commercial truth

Principle 2

Jobs change infrastructure. APIs create intent.

APIs request work; workers do the work.

Principle 3

Billing decides entitlement, not shell behavior.

Billing says what should happen. Provisioning/ops executes it.

Principle 4

Every privileged action must be auditable.

No invisible support magic.

Principle 5

Tenant admin experience should feel safe and constrained.
